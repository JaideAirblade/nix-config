# AD Test Lab Implementation Plan

> **For Hermes:** Use subagent-driven-development skill to implement this plan task-by-task.

**Goal:** Declarative Windows AD test lab on NixOS — persistent domain controller VM + throwaway Windows client VMs that can be spun up, joined to the domain, tested, and nuked.

**Architecture:** libvirt + QEMU/KVM with declarative VM definitions in NixOS. The domain controller (Windows Server) persists with a base snapshot. Client VMs (Windows 10/11) are created from a base image, joined to the domain via PowerShell, tested, then reverted to snapshot or deleted. An isolated libvirt network keeps the lab separate from the host network. Justfile targets provide `just lab-up`, `just lab-down`, `just lab-fresh-client`.

**Tech Stack:** libvirt, QEMU/KVM, NixOS virtualisation modules, PowerShell (in-VM), Windows Server Eval ISO, Windows 10/11 Eval ISO

---

## Current Context

- **Host:** UwU — 32 cores, 62GB RAM, 3.3TB free disk, KVM available, nested virt enabled
- **No existing libvirt/virt-manager** — clean slate
- **NixOS config:** dendritic/flake-parts pattern, modules in `modules/`, hosts in `hosts/`
- **This module will be shared** — both UwU and OwO-Family want it (not TSBW-W01800 — work laptop, limited resources)

## Assumptions

- You have (or can download) Windows Server 2022 Eval ISO + Windows 10/11 Eval ISO
- The domain controller is set up once (manually, inside the VM) and then snapshotted
- Client VMs use sysprep'd base images for fast domain-join workflows
- The lab network is isolated (NAT, not bridged) — VMs get DHCP from libvirt, not the real network
- You want this on UwU and OwO-Family (both have the resources)

**Step 1: Create the module**

`modules/virtualisation/default.nix`:

```

`modules/virtualisation/libvirt.nix`:
```nix
# libvirt + QEMU/KVM virtualisation for running Windows VMs.
#
# Provides:
# - libvirtd service (QEMU driver)
# - virt-manager (GUI for managing VMs)
# - virsh CLI
# - Default storage pool at /var/lib/libvirt/images
# - jaide added to libvirtd group (no sudo needed for virsh)
# - Nested virtualisation enabled (for VMs-in-VMs if needed)
{ pkgs, lib, config, ... }:

{
  virtualisation.libvirtd = {
    enable = true;
    qemu = {
      package = pkgs.qemu_kvm;
      ovmf.enable = true;  # UEFI firmware for VMs
      swtpm.enable = true;  # TPM emulation (Windows 11 needs this)
    };
  };

  # GUI VM manager
  programs.virt-manager.enable = true;

  # CLI tools
  environment.systemPackages = with pkgs; [
    virt-manager
    virt-viewer
    bridge-utils
    libguestfs  # virt-customize, virt-sysprep for image manipulation
  ];

  # Add jaide to libvirtd group so virsh works without sudo
  users.users.jaide.extraGroups = [ "libvirtd" ];

  # Enable nested virt for AMD (UwU) — Intel hosts set this differently
  boot.kernelModules = lib.optionals config.hardware.cpu.amd.updateMicrocode [
    "kvm-amd"
  ];
  boot.extraModprobeConfig = ''
    options kvm-amd nested=1
  '';
}
```

## Task 2: Create the isolated lab network module

**Objective:** Create an isolated NAT network for the AD lab so VMs can talk to each other but don't touch the host's real network.

**Files:**
- Create: `modules/virtualisation/ad-lab.nix`

**Step 1: Create the lab network definition**

`modules/virtualisation/ad-lab.nix`:
```nix
# AD test lab — isolated libvirt network + VM definitions.
#
# Network: 192.168.100.0/24 (NAT, isolated from host network)
#   - DHCP: 192.168.100.100-200 (for client VMs)
#   - Static: 192.168.100.10 (domain controller, set manually)
#
# VMs:
#   - dc1: Windows Server 2022 (domain controller, persistent)
#   - client1..N: Windows 10/11 (throwaway, revert to snapshot)
#
# The lab is entirely declarative — `just lab-up` creates the network +
# starts the DC, `just lab-down` tears it all down.
{ pkgs, lib, ... }:

let
  labNetName = "ad-lab";
  labNetIP = "192.168.100.0/24";
  labDhcpStart = "192.168.100.100";
  labDhcpEnd = "192.168.100.200";
  dcIP = "192.168.100.10";
in
{
  # Declarative libvirt network for the AD lab
  virtualisation.libvirtd = {
    enabledN networks = lib.mkForce [ ];  # don't auto-start networks
  };

  # Script to create/start/stop the lab network
  environment.systemPackages = [
    (pkgs.writeShellScriptBin "lab-net-create" ''
      set -e
      if ! virsh net-info ${labNetName} >/dev/null 2>&1; then
        echo "Creating AD lab network ${labNetName}..."
        cat <<EOF | virsh net-define /dev/stdin
      <network>
        <name>${labNetName}</name>
        <forward mode="nat"/>
        <ip address="${dcIP}" prefix="24">
          <dhcp>
            <range start="${labDhcpStart}" end="${labDhcpEnd}"/>
          </dhcp>
        </ip>
      </network>
      EOF
      fi
      virsh net-start ${labNetName} 2>/dev/null || true
      virsh net-autostart ${labNetName} 2>/dev/null || true
      echo "Lab network ${labNetName} is up (192.168.100.0/24)"
    '')
    (pkgs.writeShellScriptBin "lab-net-destroy" ''
      virsh net-destroy ${labNetName} 2>/dev/null || true
      virsh net-undefine ${labNetName} 2>/dev/null || true
      echo "Lab network ${labNetName} destroyed"
    '')
  ];
}
```

## Task 3: Add Justfile targets for lab management

**Objective:** Convenient commands for spinning up/tearing down the lab.

**Files:**
- Modify: `Justfile`

**Step 1: Add lab targets**

```makefile
# ── AD test lab ─────────────────────────────────────────────────
# Start the AD lab network + domain controller VM
lab-up:
  lab-net-create
  virsh start ad-dc1 2>/dev/null || echo "DC not defined yet — create it with: just lab-create-dc"
  virsh start ad-client1 2>/dev/null || echo "No client VMs running"

# Stop the entire lab (network + all VMs)
lab-down:
  for vm in $(virsh list --name --state-running 2>/dev/null | grep '^ad-'); do virsh shutdown $vm; done
  sleep 3
  for vm in $(virsh list --name --state-shutoff 2>/dev/null | grep '^ad-'); do virsh destroy $vm 2>/dev/null || true; done
  lab-net-destroy

# List lab VMs + their state
lab-status:
  echo "=== Network ===" && virsh net-info ad-lab 2>/dev/null || echo "network not running"
  echo "=== VMs ===" && virsh list --all 2>/dev/null | grep '^ad-' || echo "no lab VMs"

# Create a fresh client VM from base image (throws away old one)
lab-fresh-client name="ad-client1":
  -virsh destroy {{name}} 2>/dev/null || true
  virsh undefine {{name}} --remove-all-storage 2>/dev/null || true
  echo "Creating fresh VM {{name}} from base image..."
  virt-clone --name {{name}} --original ad-client-base --auto-clone
  virsh start {{name}}
  echo "VM {{name}} created and started. Use 'virsh console {{name}}' or RDP to connect."

# Revert a client VM to its base snapshot (faster than fresh clone)
lab-revert name="ad-client1":
  virsh snapshot-revert {{name}} base-snapshot
  virsh start {{name}}
  echo "VM {{name}} reverted to base snapshot and started."
```

## Task 4: Create the domain controller VM definition script

**Objective:** Script to create the DC VM from the Windows Server ISO (one-time setup).

**Files:**

- Create: `modules/virtualisation/scripts/create-client-base.sh`

**Step 1: DC creation script**

`create-dc.sh` — creates the Windows Server VM, boots from ISO:
```bash
#!/usr/bin/env bash
set -euo pipefail
VM_NAME="ad-dc1"
ISO="${1:-$HOME/Downloads/Windows_Server_2022_EVAL_x64.iso}"
DISK_SIZE="60G"
RAM="4096"
VCPUS="4"

if virsh dominfo "$VM_NAME" >/dev/null 2>&1; then
  echo "VM $VM_NAME already exists"
  exit 0
fi

echo "Creating DC VM $VM_NAME..."
virt-install \
  --name "$VM_NAME" \
  --network network=ad-lab \
  --ram "$RAM" \
  --vcpus "$VCPUS" \
  --os-variant win2k22 \
  --disk "size=$DISK_SIZE,bus=virtio,format=qcow2" \
  --cdrom "$ISO" \
  --graphics spice \
  --boot uefi \
  --noautoconsole \
  --wait -1

echo "DC VM $VM_NAME created. Install Windows Server, promote to DC, then run:"
echo "  virsh snapshot-create-as $VM_NAME base-snapshot"
```

**Step 3: Domain-join PowerShell script**

`domain-join.ps1` — run inside the client VM to join the domain:
```powershell
# Join the AD domain and place in a specified OU.
# Usage (from within the VM):
#   .\domain-join.ps1 -Domain "lab.local" -OU "OU=Test,DC=lab,DC=local"

param(
  [Parameter(Mandatory=$true)]
  [string]$Domain,
  
  [Parameter(Mandatory=$false)]
  [string]$OU
)

# Set static IP + DNS to point to DC
$adapter = Get-NetAdapter | Where-Object { $_.Status -eq "Up" } | Select-Object -First 1
$dcIP = "192.168.100.10"

Set-NetIPInterface -InterfaceIndex $adapter.ifIndex -Dhcp Disabled
New-NetIPAddress -InterfaceIndex $adapter.ifIndex -IPAddress "192.168.100.101" -PrefixLength 24 -DefaultGateway $dcIP
Set-DnsClientServerAddress -InterfaceIndex $adapter.ifIndex -ServerAddresses $dcIP

# Join domain
if ($OU) {
  Add-Computer -DomainName $Domain -OUPath $OU -Credential (Get-Credential) -Restart -Force
} else {
  Add-Computer -DomainName $Domain -Credential (Get-Credential) -Restart -Force
}

Write-Host "Joined domain $Domain. VM will restart."
```

# AD Test Lab

## One-time setup

1. Download eval ISOs:
   - Windows Server 2022: https://www.microsoft.com/en-us/evalcenter/evaluate-windows-server-2022
   - Windows 10/11: https://www.microsoft.com/en-us/software-download/

2. Start the lab network:
   ```bash
   just lab-up
   ```

3. Create the domain controller (one-time):
   ```bash
   just lab-create-dc iso=~/Downloads/Server_2022.iso
   # → Windows Server installer boots in virt-manager
   # → Install Windows Server, set IP to 192.168.100.10
   # → Promote to Domain Controller (Server Manager → AD DS → Promote)
   # → Set domain name (e.g. lab.local)
   ```

4. Snapshot the DC after setup:
   ```bash
   virsh snapshot-create-as ad-dc1 dc-base-snapshot
   ```

5. Create the client base image (one-time):
   ```bash
   just lab-create-client-base iso=~/Downloads/Win11.iso
   # → Windows installer boots
   # → Install Windows 11, enable RDP, run sysprep
   ```

6. Snapshot the client base:
   ```bash
   virsh snapshot-create-as ad-client-base base-snapshot
   ```

## Daily workflow

### Start the lab
```bash
just lab-up
# → starts network + DC VM
```

### Spin up a fresh client
```bash
just lab-fresh-client name=ad-client1
# → clones from base image, starts VM
# → RDP to 192.168.100.101
```

### Join client to domain
```powershell
# Inside the client VM (via RDP):
# Copy domain-join.ps1 from \\192.168.100.10\share or manually
.\domain-join.ps1 -Domain "lab.local" -OU "OU=Test,DC=lab,DC=local"
# → enters domain admin credentials, joins domain, restarts
```

### Test stuff
- GPOs, PowerShell scripts, registry changes, software deployment
- Anything you'd test on a real AD client

### Nuke the client (start fresh next time)
```bash
just lab-fresh-client name=ad-client1
# → destroys old VM, creates fresh clone from base
```

### Or revert to snapshot (faster)
```bash
just lab-revert name=ad-client1
# → reverts to base snapshot, starts fresh
```

### Shut it all down
```bash
just lab-down
# → stops all lab VMs, destroys network
```

## Updating the base images
When you need a newer Windows version or patches:
1. Start the base VM: `virsh start ad-client-base`
2. Windows Update, install tools, configure as desired
3. Sysprep: `sysprep /generalize /oobe /shutdown`
4. Update snapshot: `virsh snapshot-delete ad-client-base base-snapshot && virsh snapshot-create-as ad-client-base base-snapshot`
```

## Risks and Tradeoffs

1. **Disk space** — Each VM uses 40-60GB. DC (60GB) + 1-2 clients (40GB each) = ~140GB. UwU has 3.3TB free so fine. OwO-Family depends on the disk.

2. **RAM** — DC (4GB) + 1 client (4GB) = 8GB. UwU has 62GB. OwO-Family needs checking.

3. **Manual Windows setup** — The DC and client base images require manual Windows installation (one-time). This can't be fully automated without unattended install scripts (Windows autounattend.xml). Could add later.

4. **OwO-Family resources** — The 750 Ti + old Intel CPU might not handle VMs well. Check RAM before enabling virtualisation there. Maybe only on UwU for now.

5. **Network isolation** — The lab uses NAT (192.168.100.0/24), isolated from the real network. VMs can reach the internet (NAT) but the host's network is not exposed to the VMs and vice versa.

6. **virt-clone vs snapshot revert** — `lab-fresh-client` does a full clone (slow, uses disk). `lab-revert` reverts a snapshot (fast, no extra disk). Prefer `lab-revert` for quick testing; use `lab-fresh-client` when the snapshot is corrupted.

## Open Questions

- Do you want unattended Windows install (autounattend.xml) for fully automated VM creation? More complex but no manual Windows setup.
- Do you want multiple client VMs at once (ad-client1, ad-client2, ...)? The scripts support it — just pass different names.
- Should the lab network bridge to the real network (so you can RDP from your phone/tablet)? Currently NAT-only. Bridging is possible but has security implications.
- Do you want the DC to auto-start on boot? Currently manual (`just lab-up`).
